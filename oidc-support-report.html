Grounded in nixos-config, Jasper host

Adding OIDC support to jasper.jentek.dev

This report is based on the current repository state for nixos-systems/jasper. It assumes the identity provider would be exposed as oidc.jentek.dev. The main target is Tailscale custom OIDC authentication, using cjen1@jentek.dev as the login identity.

Generated 2026-07-03. External project details were checked against current upstream documentation where practical.

Executive recommendation

Recommended path: run Dex as a NixOS-native service on Jasper, expose it behind Caddy at oidc.jentek.dev, and register a single Tailscale OIDC client with callback URL https://login.tailscale.com/a/oauth_response.

Dex aligns well with this goal because Tailscale only needs a standards-compliant OIDC provider, not a full app portal, reverse-proxy auth gateway, SCIM suite, or policy UI. The local NixOS channel already includes services.dex, and Tailscale's current custom OIDC docs list Dex as needing no extra provider-specific configuration.

The main operational requirement is not Dex itself; it is stable issuer discovery. Tailscale requires WebFinger at the email domain and the issuer in WebFinger must exactly match Dex's /.well-known/openid-configuration issuer. Serving jentek.dev from Cloudflare is the missing piece that makes this work.

Target architecture

Identity provider

Runs on Jasper as services.dex. Public issuer URL: https://oidc.jentek.dev.

Caddy

Terminates HTTPS, proxies Dex, and serves the WebFinger JSON needed by Tailscale at the email domain.

Tailscale

Configured as one Dex static client using scopes openid profile email and callback https://login.tailscale.com/a/oauth_response.

Treat Dex as the root authentication path for your tailnet. Once Tailscale depends on it, the issuer hostname, WebFinger endpoint, Dex storage, signing keys, and admin recovery path become critical infrastructure.

Tool options and tradeoffs

ToolFit for this serverStrengthsCosts and risks
Dex Best fit for Tailscale custom OIDC on this server. NixOS module exists as services.dex. Small OIDC provider, declarative static clients, good for brokering login to a local password DB or an upstream connector. Tailscale's custom OIDC docs list Dex with no extra provider-specific configuration. Minimal UI and limited user lifecycle management. It is not an app portal or reverse-proxy auth gateway. You must provide WebFinger, durable storage, secrets, and a recovery path yourself.
Kanidm Strong NixOS-native alternative if you later want identity management, not just Tailscale login. Packaged as services.kanidm; supports OIDC/OAuth2 client integrations; can also integrate with PAM/NSS later. Less broad app-proxy ecosystem than Authentik. Some setup is CLI-driven. Reverse-proxy auth for non-OIDC apps needs an additional component such as oauth2-proxy or app-native OIDC.
Authentik Better if you want an app portal, rich policies, and proxy auth for many web apps. OIDC provider, proxy provider/outposts, LDAP/SAML options, good UI, app-level policy model. Good choice for managing many self-hosted apps. More moving parts: server, worker, PostgreSQL, Redis. Heavier than needed if the real target is only Tailscale auth.
Authelia Good if the primary goal is protecting Caddy routes with login and MFA. NixOS module exists locally as services.authelia.instances. Caddy has a built-in forward_auth directive, and Authelia documents Caddy integration. Authelia also has OIDC provider support. Historically strongest as an auth gateway rather than full IAM. Multi-domain protection may require multiple instances according to the NixOS module text. OIDC admin UX is less polished than Authentik.
oauth2-proxy Useful companion, not a full identity provider. Can sit in front of apps and delegate login to Authentik, Kanidm, Dex, Keycloak, Google, GitHub, etc. Official docs cover generic OIDC provider configuration. Adds another proxy/session layer. It solves "protect this upstream," not "manage identities." Caddy can often call an auth gateway directly via forward_auth.
Keycloak Technically solid but likely too heavy here. Mature enterprise IAM, realms, clients, groups, SAML/OIDC depth. NixOS module exists. Higher operational complexity, Java service footprint, more concepts than a personal Jasper server probably needs.
Pocket ID / ZITADEL Worth considering only for specific preferences. Pocket ID emphasizes passkeys and simplicity. ZITADEL is polished and cloud-native. Less aligned with the existing repo than Authentik/Kanidm/Authelia. Validate packaging and backup story before adopting.

How to add it here

1. Decide and reserve the issuer host

Use one canonical issuer. OIDC clients validate issuer URLs exactly, and Tailscale says changing issuer later requires updating WebFinger and contacting support. I would choose https://oidc.jentek.dev.

2. Add a Jasper Dex module

Dex can run as a NixOS service rather than an Arion container. Use local PostgreSQL storage once this gates your tailnet. Use an Agenix-backed file for the Tailscale client secret. 

# nixos-systems/jasper/default.nix
imports = [
  # ...
  ./dex.nix
];
# sketch only: nixos-systems/jasper/dex.nix
{ config, ... }: {
  imports = [
    ../../secrets
  ];

  services.dex = {
    enable = true;
    settings = {
      issuer = "https://oidc.jentek.dev";
      storage = {
        type = "postgres";
        config.host = "/var/run/postgresql";
      };
      web.http = "127.0.0.1:5556";
      oauth2.skipApprovalScreen = true;
      enablePasswordDB = true;

      staticClients = [
        {
          id = "tailscale";
          name = "Tailscale";
          redirectURIs = [ "https://login.tailscale.com/a/oauth_response" ];
          secretFile = config.age.secrets."dex-tailscale-client-secret".path;
        }
      ];

      staticPasswords = [
        {
          email = "cjen1@jentek.dev";
          username = "cjen1";
          userID = "cjen1";
          # Generate with: dex hash-password
          hash = "$2a$10$REPLACE_WITH_BCRYPT_HASH";
        }
      ];
    };
  };

  services.postgresql = {
    enable = true;
    ensureDatabases = [ "dex" ];
    ensureUsers = [
      {
        name = "dex";
        ensureDBOwnership = true;
      }
    ];
  };

  services.caddy.virtualHosts."oidc.jentek.dev".extraConfig = ''
    reverse_proxy 127.0.0.1:5556
  '';
}
The snippet above is a structure sketch, not a drop-in final module. The client secret must exist as an age secret, and the password hash should be generated with Dex tooling.

3. Serve WebFinger from the email domain

Tailscale discovers the issuer using the email domain, not the issuer subdomain. For cjen1@jentek.dev, serve https://jentek.dev/.well-known/webfinger with a JRD whose issuer exactly equals Dex's issuer. 

{
  "subject": "acct:cjen1@jentek.dev",
  "links": [
    {
      "rel": "http://openid.net/specs/connect/1.0/issuer",
      "href": "https://oidc.jentek.dev"
    }
  ]
}

If the login domain is not yours, you cannot complete this step yourself and should stop at the Dex deployment.

4. Configure Tailscale custom OIDC

In Tailscale's custom OIDC setup, provide:

  • Issuer URL: https://oidc.jentek.dev.
  • Client ID: tailscale.
  • Client secret: the value stored in dex-tailscale-client-secret.
  • Scopes: openid profile email.
  • Callback URL: already fixed by Tailscale as https://login.tailscale.com/a/oauth_response; it must be in Dex's redirectURIs.

Security and operational checklist

  1. Pick one permanent issuer hostname before configuring clients.
  2. Add DNS for the issuer hostname to Jasper, and add the matching Caddy vhost.
  3. Serve WebFinger at the domain part of the Tailscale login email.
  4. Put the Dex Tailscale client secret into Agenix.
  5. Generate the Dex password hash with dex hash-password; do not store a cleartext password in Nix.
  6. Back up the Dex PostgreSQL database before switching the tailnet to it.
  7. Keep an SSH recovery path to Jasper that does not depend on Tailscale auth.
  8. Document the exact issuer, client ID, callback URL, and WebFinger response before completing Tailscale setup.
